## 0. 一句话项目介绍（90 秒版）

> 我用「Claude 规划 / DeepSeek 执行 / Codex 测试修复 / 我做最终验收」的**多 AI agent 流水线**，开发了一个多 Agent 编排框架 `campaign`：核心是**事件溯源**（状态由事件回放派生、按 run_id 隔离）+ **A2A 协议**（本地/HTTP/SSE 可换传输）+ **全链路治理**（Policy-as-Code 闸）+ **抗毁性**（熔断/协同制动/三级减员）+ **三层记忆**（短/中/长 RAG）+ **跨节点分布式 MVP**，零核心外部依赖、172 个测试。过程里解决了一串系统/并发/分布式/AI 协作的硬问题，也明确标注了没做的边界（不假装分布式一致性）。

## 1. 多 Agent 协作流水线设计（编排 / 系统设计）
- **背景**：单个模型既贵又容易在长任务里漂移。
- **做法**：固定四段流水线——**Claude 规划 → DeepSeek(便宜) 写实现 → Codex 写测试+修复 → Claude 独立验收**；每段职责在指令里写死（"你只负责测试，不要重构"）。
- **关键设计**：① 执行方**看不到聊天上下文**，所以计划必须**自包含**（含"先读哪些文件确认真实签名"+ 验收命令）；② **「便宜的干活、贵的把关，把关不降级」**——质量门禁不下放给便宜模型。
- **结论**：多 agent 的价值不在"谁更强"，在**职责边界 + 验证闭环**；像分布式系统一样设计角色与不变量。

## 2. 诊断"静默失败"的桥接（调试 / 根因定位）
- **问题**：调度 Codex 的 MCP 桥接突然零输出——`status` 卡 `stale`、`stdout/stderr` 全 0 字节、无 exit code、无产物。
- **排查**：比对时间戳发现 **MCP node 进程启动于 07:07、而桥接代码 07:30 才修过** → 进程跑的是旧代码（Node 不热重载）；旧代码用 npm 的 `codex.ps1` shim、把 12KB 计划当位置参数且不重定向 stdio → shim re-spawn 后 detach，输出全丢。
- **解法**：重启 MCP 加载新代码；新代码检测 `.ps1` shim 改用 `node codex.js` 并把 stdin/stdout/stderr 重定向到文件、计划走 stdin。
- **原理**：进程在加载时对代码取快照、不热重载；"能连上但没数据"要往**进程生命周期 + 子进程 detach/IO 重定向**方向查。

## 3. Windows 沙箱 `spawn setup refresh`（操作系统 / 排障手法）
- **问题**：Codex 每条命令 `windows sandbox: spawn setup refresh`，exit -1 in 0ms。
- **根因**：工具升级后沙箱后端被设成 `elevated`（要 UAC 提权），而无头/后台启动**触发不了 UAC**，setup helper 立即 spawn 失败。
- **巧排查**：故意传非法值 `-c windows.sandbox="__bogus__"` → 报错**枚举出合法值** `elevated|unelevated`。
- **解法**：改 `unelevated`（受限令牌沙箱，无头可用且仍隔离）。
- **结论**：① 提权上下文 vs 无头进程是经典坑；② "用非法值逼出合法枚举"是快速摸接口的实用技巧；③ 升级会重写配置，要可复查。

## 4. 验证纪律：不信自述（质量意识）★ 面试加分
- **问题**：embedded Python 经 agent shell 跑 pytest **不打印 summary**；执行方只看 exit code 0 就报"全过"，实则有 `KeyError`、有用错 key（`agent_id` 应为 `actor`）。
- **解法**：验收阶段**始终 Claude 独立复跑** `pytest` + 用 `--junitxml` 看真实计数 + **逐条查 `FAILED`**；多次抓出便宜执行方漏检的真 bug（`ask()` 用 `Part` 没 import 的 NameError 等）。
- **原理**：**Trust but verify**；exit code 不等于"无失败"；自动化产出必须有独立校验层。

## 5. 事件溯源 + run_id 隔离（架构）
- **设计**：不存可变状态，全部动作 append 成事件，当前态 = `derive_state(events, run_id)` 回放派生。
- **收益**：可回放、可审计、并发安全；天然支持**跨进程/换设备恢复**（HITL 暂停的任务，另一台机 `git pull` 同一事件库即可 `resume`）；`run_id` 让多次运行互不混流。
- **结论**：事件溯源把"状态一致性"问题转成"事件顺序"问题，是 agent 系统做可恢复/可审计的高性价比选择。

## 6. A2A 协议 + 传输可换（接口设计 / 分布式）
- **设计**：定义 Message/Part/AgentCard 协议 + `Transport` 抽象；`InProcess` ↔ `HTTP/JSON-RPC` ↔ 流式 SSE 只换实现、**不动编排**。能力发现（`agent/cards`）让协调端自动注册远端。
- **硬点**：HTTP 传输加**重试+指数退避**后，必须配**幂等去重**（按 message_id 缓存），否则重试会让远端 worker **重复执行**。
- **结论**：依赖倒置（编排依赖抽象传输）+ **幂等是重试的前提**——是分布式调用的两条铁律。

## 7. 真并发 + 治理闸的并发安全（并发）
- **问题**：把串行改成 `gather` 并发后，多个任务并发读改写同一个**预算账本**会竞争。
- **解法**：用 `PolicyGate`（Governor + `asyncio.Lock`）把"花费记账"收口、串行化关键段；默认并发=1 保持兼容。
- **附带洞察**：审查时发现一批安全机制"**接了线但没被真实信号驱动**"——预算 vet 的 cost 恒为 0、减员的 health 没人更新——等于形同虚设，逐个接上真实信号才生效。
- **结论**：asyncio 单线程也有跨 `await` 的竞态；"看起来接了"不等于"真在起作用"，要顺着数据流验证。

## 8. 信任边界 / 不可信内容（安全）
- **做法**：A2A 消息**发送方 allowlist 防伪**、检索/跨 agent 内容标 `untrusted`、**注入扫描**（"ignore previous instructions"等模式）、敏感数据强制本地不出域。
- **结论**：**外部/工具/agent 返回的内容一律当数据、不当指令**——这是 agent 安全的第一原则（prompt injection 防线）。

## 9. 诚实的工程边界（判断力）★ 面试加分
- **取舍**：分布式一致性需要 Postgres/Redis + 真实集群才能测；没有就**不假装做**——只写 `SCALING.md` 标清"哪些是 MVP、哪些仍每进程"，不在测试里伪造"已具备一致性"。
- **结论**：**诚实标注未实现边界，胜过假装完成**；可测性是"做没做"的判据。

## 10. 跨设备上下文记忆（工程实践）
- **做法**：把易失的"对话上下文"沉淀成仓库文件（`WORKLOG.md` 带日期+执行 agent 归属、`CLAUDE.md` 记忆协议），git 同步——换设备 `git pull` + 读 WORKLOG 即续上。
- **结论**：agent 协作的"记忆"要落成**版本化的文件**，不依赖某次会话。

## 高频追问 · 应答预案

- **Q：多 agent 比单 agent 强在哪？** A：不在单点更强，在**职责分离 + 成本分层 + 验证闭环**——便宜模型批量执行、贵模型规划与把关，像团队而非单兵；代价是要设计好接口与不变量。
- **Q：怎么保证便宜模型的产出质量？** A：把关不下放——独立复跑测试、查 FAILED、抽查实现；它产过 NameError/错 key，都在验收层拦下。
- **Q：事件溯源不会爆吗？** A：会，所以按 run_id 隔离 + 派生视图缓存；高并发写需换共享有序日志（Postgres/Kafka），已在 SCALING 里标明。
- **Q：重试为什么要幂等？** A：网络重试可能让同一请求到达两次，worker 会重复执行/重复扣预算；按 message_id 去重保证"至多一次生效"。
- **Q：asyncio 单线程还要锁？** A：要——竞态发生在 `await` 切换点；读改写共享状态跨 await 必须用 `asyncio.Lock`。
- **Q：为什么不做完分布式？** A：没有可测的真实底座时硬做 = 自欺；我选择标清边界 + 留可替换接口（EventLog/Transport/Gate 都是抽象），需要时换实现即可。
